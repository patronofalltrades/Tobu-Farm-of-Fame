## Findings

### Minor ‚Äî Roster dropdown opens on mount (not "closed by default")

**File:** `src/components/RosterPicker.tsx`
`autoFocus` on the search input fires an `onFocus` handler that sets `isOpen(true)` immediately on mount, so the dropdown is visibly open the instant the picker renders ‚Äî P0-4's literal "closed by default" acceptance criterion doesn't strictly hold. This is arguably better UX (zero extra tap for a first-time 70-name picker) and was very likely an intentional tradeoff, not an oversight. Flagging for awareness, not requesting a change.

### Low severity, not user-reachable ‚Äî Overlay state has no mutual exclusion

**Files:** `src/App.tsx`, `src/scene/BullHerd.tsx`
`selectedTobuId`, `isMascotOpen`, `isBarnOpen`, `isLeaderboardOpen`, `isAdminPinOpen`, and `isAdminPanelOpen` are independent booleans with no "only one overlay at a time" invariant enforced. During this pass's automated testing, dispatching synthetic click events directly on the `canvas` element (bypassing the browser's real hit-testing/occlusion) caused a Tobu bubble, the mascot bubble, and the Admin PIN gate to all become simultaneously `true`, stacking three `.speech-bubble`-styled overlays on top of each other ‚Äî invisible until the topmost was dismissed.

**This is not reachable by a real user.** Once any overlay's `position:fixed inset:0` backdrop is on screen, it intercepts all further pointer events ‚Äî a real tap can't "reach through" to trigger a second landmark or the topbar underneath. The only reason this surfaced here is that direct `canvas.dispatchEvent(...)` calls skip the DOM's actual point-in-time occlusion check.

Worth noting as architectural hygiene (a single `activeOverlay` enum instead of 6 independent booleans would make this invariant explicit and remove the latent fragility), but does not block ship ‚Äî no fix applied this pass.

## Testing Methodology Notes (for future QA passes)

- **R3F/canvas raycast targeting requires `offsetX`/`offsetY` to be explicitly defined via `Object.defineProperty`** on synthetic `PointerEvent`s ‚Äî these are normally read-only computed properties, and default to `0` on manually-constructed events, causing R3F's raycaster to miss every time. This exact gotcha was already flagged in `e2e-report-2026-07-12.md`'s Environment notes and cost significant time to rediscover this pass ‚Äî future passes should reuse the `fireAt()` helper pattern established in this report's tool calls directly.
- **`preview_click(selector)` is unreliable against small/moving targets inside a WebGL canvas** (bulls wander every frame, landmarks are small). A dense grid-sweep with `Object.defineProperty`-patched offsets, checking for a distinguishing DOM marker after each dispatch (e.g. `.reaction-row` to confirm a *Tobu* bubble specifically, not the mascot's), was the only reliable method found.
- **Reads immediately following a state-changing click can race React's async commit.** Two "failures" this pass (roster confirm button, reaction toggle) were purely `preview_eval` checking DOM state in the same or an immediately-following call, before React/Zustand had flushed. A ~300-500ms `setTimeout` after any click before asserting state eliminated this reliably.
- **rAF-based fps sampling doesn't work in this hidden-tab preview environment** ‚Äî confirmed again this pass (4 frames in 8 seconds). Don't re-attempt continuous sampling; rely on qualitative screenshot evidence or a different profiling approach if precise fps numbers are ever required.
